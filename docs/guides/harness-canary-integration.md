# Harness + Canary Integration Guide

Canary is both a standalone Claude Code plugin **and** a harness-engineering
project. When a developer runs **both** the canary plugin and the harness
toolkit, the same test-intelligence skills surface through two routers and
several config/state surfaces overlap. This guide is the single source of truth
for **which entry point to reach for**, **which file owns which setting**, and
**what actually gates a merge**.

It resolves the routing/ownership ambiguity tracked in issues #308 and #311.

---

## 1. Skill routing and double-registration

### Why a skill shows up twice

Install both the canary plugin and harness and each canary test-intelligence
skill registers **twice**:

| Surface                  | Example name              | Where it comes from                                                                                                                                  |
| ------------------------ | ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Plugin native command    | `/canary-ci-ready`        | `commands/canary-ci-ready.md` — a first-class Claude Code slash command shipped in the plugin manifest.                                              |
| Harness-surfaced command | `harness:canary-ci-ready` | `agents/commands/claude-code/harness/canary-ci-ready.md`, generated by `harness generate-slash-commands` from the skill's `SKILL.md` + `skill.yaml`. |

Both paths ultimately drive the **same** underlying workflow — the plugin
command routes through the canary agent, the harness command reads the skill's
`SKILL.md` directly — so there is **no behavioural divergence**. Neither "wins";
they are two doors into one room.

**Which to prefer:**

- **Invoking directly?** Use the plugin's `/canary-X` command. It is the
  canonical, argument-hinted entry and does not depend on the harness router
  being installed.
- **Letting the harness router intent-match and dispatch?** It will reach the
  skill as `harness:canary-X`. That is expected; it resolves to the same skill.

> The `harness:canary-X` commands and `.harness/skills-index.json` are
> **generated artifacts**. Their metadata is derived from each skill's
> `skill.yaml` (`name`, `description`) via `harness generate-slash-commands`. Do
> **not** hand-edit the generated command files or the (gitignored) index — edit
> `skill.yaml` and regenerate. A stale generation is what left the five surfaced
> skills advertised as the placeholder "Test intelligence skill"; the fix was to
> populate `skill.yaml` descriptions and regenerate (#308).

### Disambiguation matrix — canary skill vs harness skill

Many canary and harness skills overlap in intent. Reach for the **canary** skill
when the job is _test-suite-shaped_ (a suite, a framework, a test's blast
radius). Reach for the **harness** skill when the job is _codebase/graph-shaped_
(architecture, whole-repo impact, release readiness).

| Canary skill                                  | Harness counterpart                                | Reach for canary when…                                                                                                                                        | Reach for harness when…                                                                                                                  |
| --------------------------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `canary-failure-impact`                       | `harness:impact-analysis`                          | You want the **blast radius of a failing/undetected test** on a specific function or code path, with a test-severity label.                                   | You want the **graph-wide "if I change X, what breaks?"** across modules and dependents, independent of tests.                           |
| `canary-critical-areas`                       | `harness:test-advisor`, `harness:hotspot-detector` | You want a **ranked list of where to add tests** (churn × dependents × business-critical × coverage depth) with recommended **test types**.                   | You want structural **risk hotspots** (co-change/churn) or the harness graph's **"what should I run / what's untested"** coverage audit. |
| `canary-ci-ready`                             | `harness:verify`, `harness:release-readiness`      | You want a **test-suite readiness score** (coverage depth, flakiness, assertion quality, critical-path coverage, runtime).                                    | You want the **binary pass/fail quick gate** (`verify`) or a **npm/release-publish readiness** audit (`release-readiness`).              |
| `canary-generate-test` / `/canary-write-test` | `harness:tdd`                                      | You want the **AI to write the test** from a natural-language description (CLI batch or session).                                                             | You want the **TDD discipline loop** — write a failing test first, watch it fail, then implement. See the composition pattern below.     |
| `/canary-review-test`                         | `harness:test-craft`, `harness:code-review`        | You want a **test-specific review** (brittleness, anti-patterns, coverage gaps) and/or to **promote** a generated test into the committed suite.              | You want the **8-axis LLM test-quality critique** (`test-craft`) or a **multi-persona code review** of the diff.                         |
| `canary-edge-case-discovery`                  | `harness:test-craft`                               | You want to **surface untested edge cases** across six categories from a feature/function/suite.                                                              | You want a **quality critique of tests that already exist**.                                                                             |
| `canary-setup-harness`                        | `harness:initialize-*`                             | You are **onboarding an existing canary fork** onto the harness guardrails (installs CLI, writes `harness.config.json`, wires CI, seeds the security ledger). | You are **scaffolding a brand-new harness project** or test-suite from scratch (`initialize-project`, `initialize-test-suite-project`).  |

**Rule of thumb:** if the noun is a _test_, a _suite_, or a _framework_, start
with canary. If the noun is a _module_, the _graph_, the _release_, or the
_architecture_, start with harness.

### The recommended composition (they cooperate)

Canary and harness are complementary, not competing. The full test-dev flow
chains both (this is the pattern already documented in `AGENTS.md`):

```text
harness:tdd          → discipline: write the failing test first (RED)
/canary-write-test   → generation: AI writes the test from your description
harness:test-craft   → quality: 8-axis audit of the generated test
/canary-review-test  → promotion: move tests/generated/ → committed suite
```

## 2. Config and state ownership

Two tools, several config/state files. This map says **what belongs where** so a
threshold or setting lands in the right file.

| File                                    | Owner     | Holds                                                                                                                                                                                        | Do **not** put here                                                                                                             |
| --------------------------------------- | --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `harness.config.json`                   | harness   | Architectural **layers**, `forbiddenImports`, complexity/coupling **thresholds**, entropy/drift config, roadmap tracker, telemetry/adoption toggles.                                         | Test-suite thresholds, per-project overlay skills, company pointers.                                                            |
| `.canary/` (runtime, **consumer-side**) | canary    | Project-local generated state and consumer config: `test-inventory.json`, `critical-areas.json`, `company.json` (org/project pointers), and the `skills/` private overlay.                   | Anything meant to live in canary's own source tree — this directory is generated in the **consuming** repo, not committed here. |
| `pyproject.toml` / `npm/package.json`   | packaging | Package version, dependencies, entry points, tool config (ruff/pytest). Version must stay in lockstep with `plugin.json` + `marketplace.json` (`test_version_consistency.py` enforces this). | Architecture rules or test-intelligence state.                                                                                  |

### Why `.canary/` does not exist in this repo (and shouldn't)

Several skills read `.canary/` paths — `canary-ci-ready` reads
`.canary/test-inventory.json`, `canary-failure-impact` and
`canary-edge-case-discovery` read `.canary/critical-areas.json`, and
company-knowledge reads `.canary/company.json`. Yet **`.canary/` is absent from
this repository — by design, not by omission:**

- `.canary/` is **gitignored** at the repo root, and
  `canary company-knowledge init` explicitly **adds** `.canary/` to a consuming
  project's `.gitignore`.
- It is a **runtime, consumer-side** directory: canary creates it inside the
  repo it is analysing to cache inventory/risk state and to hold that team's
  private overlay + company pointers.
- The **open-core boundary** depends on this: company-specific content lives
  only under `.canary/skills/` and `.canary/company.json` in a private overlay,
  never in this public tree.

**Decision (issue #311):** we **did not scaffold** a `.canary/` directory in the
repo and instead documented it as runtime/consumer-side. Scaffolding was the
higher-risk option — the directory is gitignored (a committed placeholder would
either be un-committable or require un-ignoring it, which fights the open-core
denylist guard) and an empty `.canary/skills/` could perturb overlay discovery
precedence. The references in the skills are therefore **correct**; a reader
should expect `.canary/` to be **absent until canary runs against a project**.

## 3. State ledgers: `roadmap.md` vs `CANARY_STATE.md`

Two ledgers, two owners, two jobs. They are **cross-referenced, not merged** —
keep it that way.

| Ledger                                       | Owner   | Authoritative for                                                                                                                                                                          | Updated by                                        |
| -------------------------------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------- |
| [`docs/roadmap.md`](../roadmap.md)           | harness | **Forward-looking work**: planned/in-progress/blocked features, specs, blockers, upstream dependencies. Machine-managed by harness roadmap tooling and synced to the GitHub issue tracker. | `harness roadmap` tooling + curated manual edits. |
| [`docs/CANARY_STATE.md`](../CANARY_STATE.md) | canary  | **Backward-looking activity**: a one-line append per skill run (test generated/promoted, harness set up) so a later session picks up context without re-deriving it.                       | canary skills appending log entries.              |

**How to read them together:** ask "what are we _going to do_?" → `roadmap.md`.
Ask "what did a skill _just do_ here?" → `CANARY_STATE.md`. Neither is
derivative of the other; if they disagree about the status of a piece of work,
**`roadmap.md` is authoritative for intent/status** and `CANARY_STATE.md` is a
factual activity log. Commit `fc3b90a` fixed a case where `CANARY_STATE.md` had
drifted into stale architecture narration — the lesson is to keep
`CANARY_STATE.md` to terse activity lines and leave status/planning to
`roadmap.md`.

## 4. What is the merge gate?

The **blocking** merge gate is the set of GitHub Actions workflows required on a
PR; everything else is advisory. In one breath: **`harness.yml`
(`harness ci check`) is the umbrella mechanical gate, backed by
`harness-quality.yml` (docs coverage, entropy/drift cleanup, phase-gate),
`harness-architecture.yml` (dependency layers + `validate`),
`harness-security.yml` (security scan + deps + validate), `docs-lint.yml`
(markdownlint + the removed-symbol guard), and `validate-plugin.yml`
(manifest/version consistency) — those workflows must be green to merge.**
Canary's own `canary-ci-ready` / `canary-test-pipeline` and the harness
`review-ci` / `pre-merge-brief` layers are **not** wired as required checks:
they run in the developer's Claude Code session (or as a reviewer persona) and
**inform** the merge decision — the readiness score, the multi-persona verdict,
and the senior-facing accountability brief — but do not block it. So: harness
workflows **gate**, canary/review layers **advise**.

> Known gap: `harness ci check` does not yet honour
> `entropy.analyze.drift.checkApiSignatures: false` from `harness.config.json`
> (upstream harness#838), so api-signature drift can still surface as a
> non-blocking warning in `harness.yml`. Tracked on the roadmap; findings are
> warn-severity and do not fail the gate.

## 5. The canary MCP server

Two distinct MCP surfaces mention "canary" — do not conflate them:

- **Plugin MCP server** — declared as `canary-mcp` in
  `.claude-plugin/plugin.json` (command `canary-mcp`). It exposes the six
  FastMCP tools (`canary__analyze_file`, `canary__write_test_file`,
  `canary__run_tests`, `canary__init_suite`, `canary__list_frameworks`,
  `canary__migrate`). This is the canonical canary MCP server; reference it by
  that name.
- **Harness-surfaced canary tools** — when the harness MCP server proxies
  canary's probes they resolve as `mcp__harness__canary_*` (e.g.
  `mcp__harness__canary_probe`, `mcp__harness__canary_recommend_framework`).
  These are harness-router views, not a second server.

Per the [Trusted MCP hierarchy](../../AGENTS.md#trusted-mcp-hierarchy): prefer
harness tools first, then canary (the `canary-mcp` server and the `canary-*`
skills) for anything test-related, and third-party MCP only when neither fits.
